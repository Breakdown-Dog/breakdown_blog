## 为什么要升级 HTTP？

HTTP/1.1 自 1997 年成为标准，虽然稳定但在现代 Web 场景下暴露了严重的性能瓶颈：

- **队头阻塞**：一个 TCP 连接同时只能处理一个请求
- **重复头部开销**：每次请求都发送冗余的 Header
- **低效的并发**：通常用 6 个 TCP 连接来缓解，但治标不治本

## HTTP/2 做了什么？

### 二进制分帧

HTTP/2 最大的变化是将文本协议改为 **二进制协议**。

```
HTTP/1.1 文本协议:
GET /index.html HTTP/1.1
Host: example.com

HTTP/2 二进制格式:
┌────────┬────────┬────────┬────────┐
│ Length │  Type  │ Flags  │StreamID│
├────────┴────────┴────────┴────────┤
│          Payload Data              │
└───────────────────────────────────┘
```

### 多路复用

在一个 TCP 连接上并发传输多个流（stream），**互不阻塞**：

```mermaid
HTTP/1.1:        HTTP/2:
┌─────┐          ┌─────┐
│Req 1│──→       │Req 1│──→
│Res 1│←──       │Req 2│──→
│Req 2│──→       │Res 2│←──
│Res 2│←──       │Req 3│──→
│Req 3│──→       │Res 1│←──
│Res 3│←──       │Res 3│←──
                  (交错传输)
```

### 头部压缩（HPACK）

使用 Huffman 编码 + 动态表压缩 Header，通常能减少 **80%–90%** 的头部体积。

### 服务器推送

服务器可以 **主动** 向客户端推送资源，无需客户端显式请求：

```javascript
// 服务器推送 index.html 的同时推送 style.css
// 浏览器收到后会缓存，等需要时直接使用
```

## HTTP/3 —— 基于 QUIC

HTTP/3 将传输层从 TCP 换为 **QUIC**（基于 UDP），解决了 TCP 的固有缺陷。

### 0-RTT 连接建立

```
TCP + TLS 1.3:
C ──SYN──→ S  (1-RTT)
C ←─SYN-ACK─ S
C ──ClientHello──→ S  (1-RTT)
C ←─ServerHello─── S
→ 2 RTT 才能开始发送数据

QUIC:
C ──握手+数据──→ S  (0-RTT，如果之前连接过)
C ←─握手+数据─── S
→ 首次 1-RTT，之后 0-RTT！
```

### 解决了 TCP 级队头阻塞

TCP 的队头阻塞：一个 TCP 段丢失，后续所有段都要等待重传。

QUIC 的设计：每个流独立交付，一个流的丢包 **不影响其他流**：

```
TCP 丢包:
[流1帧][流2帧][流3帧] ← 一个包丢了
[流1帧] ← 等待重传，所有流都阻塞

QUIC 丢包:
流1: [帧1][帧2]
流2: [帧1]  ← 这个包丢了
流3: [帧1][帧2][帧3]
QUIC 只重传流2的帧，流1和流3继续！
```

### 连接迁移

TCP 连接绑定 IP + 端口。切换网络（如 Wi-Fi → 4G）时，TCP 连接断开。

QUIC 使用 **Connection ID** 标识连接，IP 变化连接不断：

```
Wi-Fi:   IP=192.168.1.5  CID=ABC123
↓ 你走到电梯，切换到 4G
4G:      IP=10.0.0.42    CID=ABC123  ← 连接还在！
```

## 性能对比总结

| 特性 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|------|----------|--------|--------|
| 传输层 | TCP | TCP | QUIC (UDP) |
| 协议格式 | 文本 | 二进制 | 二进制 |
| 多路复用 | ❌ 串行 | ✅ 流级别 | ✅ 流级别 |
| 队头阻塞 | 请求级 + TCP 级 | ✅ 无请求级, ❌ TCP 级 | ✅ 完全消除 |
| 连接建立 | 2 RTT (TCP+TLS) | 2 RTT | 0-1 RTT |
| 头部压缩 | ❌ | ✅ HPACK | ✅ QPACK |
| 连接迁移 | ❌ | ❌ | ✅ |

## 现状与建议

| 场景 | 推荐协议 |
|------|----------|
| API 接口（REST/GraphQL） | HTTP/2 |
| 实时视频、游戏 | HTTP/3 |
| 物联网 / 弱网环境 | HTTP/3 |
| 内网服务 | HTTP/2（UDP 可能被防火墙拦截） |

截至 2026 年，HTTP/3 的全球支持率已超过 **85%**，新项目建议直接开启 HTTP/3。
